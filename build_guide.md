# HookKit — Build Brief for Claude Code

> **This document is an executable build brief for an autonomous coding agent.**
> It contains the full contract, a pinned tech stack, a worked reference slice, and an
> ordered, independently-verifiable milestone plan. Build to the acceptance commands.

An offline, open-source, all-in-one webhook dev + test kit for JS/TS. Three co-equal
surfaces over one shared core: a **test-native SDK**, a **dev CLI**, and a **local inspector**.

---

## 0. How to work (agent operating instructions)

1. **Read this entire brief before writing any code.** The Invariants (§1) override everything, including anything that seems more convenient.
2. **Build milestones M0 → M6 in order** (§10). Each is independently verifiable. Do not start a milestone until the previous one's acceptance command passes.
3. **Test-first for anything signing-related.** The domain gives you an objective oracle: a HookKit-generated event must verify with the provider's *own official library*. Write that golden test first (§5), then implement until green.
4. **After every task, run `pnpm verify`** (lint + typecheck + test + build). After every milestone, also run that milestone's **Acceptance** command. Never proceed on red.
5. **One source of provider truth.** All provider logic lives in `@hookkit/core`. The SDK, CLI, and inspector import it. Never duplicate signing logic into a surface package.
6. **If a task appears to require violating an Invariant, stop and surface it** in your summary instead of working around it. The Invariants are the product.
7. **Commit in small units** with Conventional Commits (`feat(core): …`, `test(sdk): …`).
8. **Prefer boring, dependency-light code.** Every new runtime dependency in `core`, `sdk`, `cli` must be justified; the security-critical signing path uses only `node:crypto`.

## 1. Invariants (non-negotiable — copy into `/CLAUDE.md`)

```md
# CLAUDE.md — project rules

## Product invariants (never break these)
- OFFLINE-FIRST: `core`, `sdk`, `cli` must function with zero network access. Their test
  suites must pass with networking disabled. No package phones home. No telemetry, ever.
- NO ACCOUNT / NO CLOUD: no API keys, no sign-up, no HookKit-operated server anywhere in
  the default paths. The ONLY network component is the optional, USER-self-hosted
  `@hookkit/relay`. Never route user traffic through a server we operate.
- RAW-BODY FIDELITY: the exact received/generated bytes (a Buffer) are the source of truth
  end to end. Never JSON.parse→JSON.stringify a payload before signing or delivery.
- PROVIDER-ACCURATE SIGNING: a generated event MUST verify against the provider's official
  verification library. This is enforced by golden tests; do not merge a provider without one.
- SIGNING USES `node:crypto` ONLY. No third-party crypto deps.
- LICENSE: MIT. Fixtures contain synthetic data only — no real PII, tokens, or secrets.

## Engineering rules
- TypeScript strict everywhere. ESM output (with CJS interop via tsup). Node >= 20.
- Provider logic lives ONLY in @hookkit/core. Surfaces import it.
- Secrets are read from env or flags, never logged, never written to disk.
- Every provider adapter must pass the shared conformance suite before it's considered done.

## Verify before done
- `pnpm verify` must be green (lint + typecheck + test + build).
```

## 4. Core contracts (implement to these interfaces verbatim)

```ts
// @hookkit/core

export interface ProviderAdapter {
  id: string;                 // 'stripe'
  displayName: string;        // 'Stripe'
  contentType: string;        // 'application/json'
  signatureHeader: string;    // 'Stripe-Signature'

  /** Compute the headers a real provider attaches to this exact raw body. */
  sign(input: { rawBody: Buffer; secret: string; timestamp?: number }): Record<string, string>;

  /** Verify a received request (powers the `verify` CLI + round-trip tests). */
  verify(input: {
    rawBody: Buffer;
    headers: Record<string, string>;
    secret: string;
    toleranceSec?: number;     // default 300
  }): { valid: boolean; reason?: string };

  events: Record<string, EventDescriptor>;
  retryPolicy: RetrySchedule;
}

export interface EventDescriptor {
  fixtureId: string;           // key into @hookkit/fixtures
  schema?: unknown;            // JSON Schema; used by fixture validation + drift detection
  apiVersions?: string[];
}

export interface RetrySchedule {
  maxAttempts: number;
  windowSec: number;
  backoff: 'exponential' | 'fixed' | 'custom';
  delaysSec?: number[];        // required iff backoff === 'custom'
}

export interface GeneratedEvent {
  rawBody: Buffer;             // sign & send THESE bytes; never re-serialize
  headers: Record<string, string>;
  parsed: unknown;             // convenience view; not used for signing
  eventType: string;
  provider: string;
}

export interface GenerateOptions {
  overrides?: Record<string, unknown>;   // deep-merged into the fixture
  apiVersion?: string;
  timestamp?: number;                     // pin for deterministic tests
  secret: string;
}

export function generate(provider: string, eventType: string, opts: GenerateOptions): GeneratedEvent;

export type DeliveryTarget = string /* URL */ | FrameworkApp /* via adapter */;
export interface DeliveryResult { status: number; body: string; headers: Record<string,string>; ms: number; }
export function dispatch(evt: GeneratedEvent, target: DeliveryTarget): Promise<DeliveryResult>;
```

SDK surface (implement in `@hookkit/sdk`):

```ts
const stripe = hookkit.stripe({ secret });          // provider-scoped builder
stripe.event('checkout.session.completed', { overrides })   // → EventBuilder
  .tamperSignature()                                  // optional mutators
  .withTimestamp(t)
  .sendTo(target)                                     // → Promise<DeliveryResult>

hookkit.harness.idempotency(builderOrEvent, target, { times })
hookkit.harness.retry(builder, target, { fail: 2, then: 200 })
hookkit.harness.ordering([...builders], target)
hookkit.harness.malformed(builder, target, { kind: 'truncated'|'badContentType'|'expiredTimestamp'|'badSignature' })

// Vitest/Jest matchers
expect(res).toBeAccepted();               // 2xx
expect(res).toHaveRejectedWithStatus(400);
```

## 5. Reference vertical slice — Stripe (build this FIRST, use as the pattern)

Implement Stripe end-to-end in M0 as the template every other provider copies.

### 5.1 Exact signing recipe

```
timestamp = opts.timestamp ?? floor(Date.now()/1000)
signedPayload = `${timestamp}.${rawBody.toString('utf8')}`
v1 = HMAC_SHA256(key=secret, msg=signedPayload) as lowercase hex
header "Stripe-Signature" = `t=${timestamp},v1=${v1}`
```
Verify: recompute with the `t` from the header; constant-time compare (`crypto.timingSafeEqual`); reject if `|now - t| > toleranceSec` (default 300).

### 5.2 One concrete fixture (`packages/fixtures/fixtures/stripe/2025-04-10/checkout.session.completed.json`)

Synthetic; expand fields later to mirror Stripe's documented shape (a fixture-completeness task, tracked by schema validation in §6).

```json
{
  "id": "evt_test_0000000000",
  "object": "event",
  "api_version": "2025-04-10",
  "created": 1710000000,
  "type": "checkout.session.completed",
  "data": {
    "object": {
      "id": "cs_test_0000000000",
      "object": "checkout.session",
      "amount_total": 2000,
      "currency": "usd",
      "customer": "cus_test_0000000000",
      "payment_status": "paid",
      "status": "complete"
    }
  }
}
```

### 5.3 The golden test to write first (`packages/core/src/adapters/stripe.golden.test.ts`)

```ts
import Stripe from 'stripe';                       // dev-dependency, used as the ORACLE only
import { generate } from '@hookkit/core';
import { expect, it } from 'vitest';

it('stripe: generated event verifies with the official stripe SDK', () => {
  const secret = 'whsec_test_secret';
  const evt = generate('stripe', 'checkout.session.completed', { secret, timestamp: 1710000000 });
  // If Stripe's own verifier accepts it, our signing is provably correct.
  const constructed = Stripe.webhooks.constructEvent(
    evt.rawBody, evt.headers['Stripe-Signature'], secret,
  );
  expect(constructed.type).toBe('checkout.session.completed');
});
```

This test class is the project's backbone: repeat it per provider using each provider's official verifier.

## 6. Other providers — exact signing recipes (each needs its own golden test)

Oracles (dev-dependencies, used only in tests): GitHub → `@octokit/webhooks-methods` (`verify`); Shopify → documented HMAC (or `@shopify/shopify-api` util); Slack → documented base-string HMAC; Standard Webhooks → `standardwebhooks`.

- **GitHub** — `X-Hub-Signature-256: sha256=` + HMAC_SHA256(secret, rawBody) hex. (Also emit legacy `X-Hub-Signature` SHA1 for completeness.) Verify with `@octokit/webhooks-methods`.
- **Shopify** — `X-Shopify-Hmac-Sha256:` = base64( HMAC_SHA256(secret, rawBody) ).
- **Slack** — base = `v0:${timestamp}:${rawBody}`; `X-Slack-Signature: v0=` + HMAC_SHA256(signingSecret, base) hex; also set `X-Slack-Request-Timestamp: ${timestamp}`. Reject in verify if timestamp older than tolerance.
- **Standard Webhooks** — headers `webhook-id`, `webhook-timestamp`, `webhook-signature`. Strip optional `whsec_` prefix from the secret, **base64-decode** the remainder to key bytes. signedContent = `${id}.${timestamp}.${rawBody}`; sig = base64( HMAC_SHA256(keyBytes, signedContent) ); header = `v1,${sig}` (space-separated list may hold multiple versions).

Fixture validation: `@hookkit/fixtures` ships a manifest `(provider, event, apiVersion) → file`; a Vitest suite validates every fixture against its adapter's `schema` and fails CI on drift.

## 7. `listen` / receiving real events — do NOT build a hosted relay

Receiving a *real* provider event on localhost needs a public path. Resolve WITHOUT operating a cloud:

1. **Default — bring-your-own-tunnel:** `hookkit listen` shells out to a tunnel the user already runs (`cloudflared`/`ngrok`/`frpc`), then captures/verifies/forwards locally.
2. **Optional — `@hookkit/relay`:** a Docker image the **user** deploys on their own host to get a stable public URL that forwards to their machine over an outbound WebSocket.

Never add a HookKit-operated endpoint. This is an Invariant, not a preference.

## 10. Milestones (ordered; each has a runnable Acceptance)

**M0 — Signing spine (Stripe).**
Scaffold monorepo, tooling, CI, `pnpm verify`. Implement `core` `SignatureEngine` + `EventGenerator` + Stripe adapter + one fixture + the §5.3 golden test.
*Acceptance:* `pnpm --filter @hookkit/core test` passes, including the Stripe-vs-official-SDK golden test, with networking disabled.

**M1 — Provider framework + breadth.**
Define `ProviderAdapter` + a **shared conformance suite** (`packages/core/src/conformance.ts`) any adapter must pass. Add GitHub, Shopify, Slack, Standard Webhooks, each with fixtures + a golden test vs its official verifier + fixture-schema validation.
*Acceptance:* `pnpm --filter @hookkit/core test` runs the conformance suite across all five providers, all green, offline.

**M2 — Test SDK + framework adapters.**
Implement `@hookkit/sdk` (builder, `sendTo`, matchers) and adapters for Express/Fastify/Next/Hono guaranteeing raw-body fidelity. Wire the four `examples/*` handlers.
*Acceptance:* `pnpm --filter "./examples/**" test` passes; each example's real signature-verification middleware runs against a HookKit-generated event and returns 200.

**M3 — CLI (offline).**
`trigger`, `replay`, `list`, `verify` with config resolution (flags → `hookkit.config.ts` → env). Secrets never logged.
*Acceptance:* start `examples/express-stripe`, run `pnpm hookkit trigger stripe checkout.session.completed --to http://localhost:3000/webhooks/stripe --secret whsec_test`; handler logs a verified event and returns 200.

**M4 — Inspector.**
Hono capture server + SQLite + React/Vite/Tailwind UI (3-pane, JSON pretty-print, signature badge, replay, forward, SSE live). `npx hookkit inspect`; Docker image.
*Acceptance:* `npx hookkit inspect` starts on `127.0.0.1:4000`; POST to a generated endpoint appears live; replay to a local URL succeeds; `docker build packages/inspector` succeeds.

**M5 — Correctness harness + `listen`.**
`harness.idempotency|retry|ordering|malformed`; `listen` with BYO-tunnel; optional `@hookkit/relay` Docker image.
*Acceptance:* harness fails a deliberately non-idempotent sample handler and passes an idempotent one; `hookkit listen 3000 --tunnel cloudflared` forwards an external POST to localhost (documented manual check).

**M6 — Fixture ecosystem.**
`hookkit fixtures add <provider> <event>` scaffolder, fixture-schema CI gate, `docs/adding-a-provider.md`. Expand event catalogs.
*Acceptance:* following `docs/adding-a-provider.md` alone, a new provider can be added and passes the conformance suite; fixture CI gate rejects an intentionally malformed fixture.

## 11. Guardrails (do NOT)

- Do **not** re-serialize payloads before signing/delivery (breaks signatures — the #1 real-world bug this tool exists to prevent).
- Do **not** add networking to `core`/`sdk`/`cli` default paths, or any telemetry/analytics/phone-home.
- Do **not** operate or hardcode a HookKit-hosted relay/tunnel endpoint.
- Do **not** duplicate provider signing logic outside `@hookkit/core`.
- Do **not** introduce a third-party crypto library; use `node:crypto`.
- Do **not** commit real secrets or non-synthetic payloads to fixtures.
- Do **not** advance past a milestone whose Acceptance command is red.
